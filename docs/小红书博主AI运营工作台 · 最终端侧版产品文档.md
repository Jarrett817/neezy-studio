# 小红书博主 AI 运营工作台・可实现版 V1 技术方案

## 1. 文档目标

这份文档用于把原始产品设想收敛成一个可开发、可验证、可发版的 V1 技术方案。

V1 的目标不是一次性做成“全能离线 AI 中台”，而是先做出一款本地优先、单账号可用、核心链路闭环的桌面工具：

- 能在本地管理素材、草稿、数据和账号风格
- 能调用本地模型完成标题/正文/标签生成
- 能把截图导入后做半自动提取，并允许人工确认
- 能在不依赖云端数据库的前提下稳定运行

## 2. V1 产品边界

### 2.1 V1 必做

- 单账号工作台
- 本地草稿管理
- 本地知识库录入、搜索、插入创作区
- 本地 AI 文案生成
- 截图导入 Beta：支持拖拽、粘贴、提取、人工确认
- 数据录入与基础复盘
- 本地配置、备份、恢复

### 2.2 V1 明确不做

- 多账号体系
- 多 Agent 协作
- 本地模型微调
- 真正意义上的“实时热点”推荐
- 承诺 8GB 设备上稳定跑多模型并发
- 承诺 OCR 98% 准确率和 1 秒内全场景提取
- 把所有模型都打进 250MB 安装包

### 2.3 V1 产品口径

V1 统一采用下面这套更真实的对外口径：

- `本地优先`，核心数据默认只保存在本机
- `离线可用`，但模型包可采用首次引导下载
- `截图半自动提取`，结果需要人工确认后入库
- `AI 辅助创作`，不是全自动运营

## 3. 技术路线总览

### 3.1 当前项目基线

基于当前仓库，建议正式统一为以下技术栈：

- 桌面容器：Tauri v2
- 前端：React 19 + React Router 7（SPA）+ TypeScript
- UI：shadcn/ui + Tailwind CSS 4 + lucide-react
- 原生层：Rust
- 数据库：SQLite
- 本地模型：Rust 侧统一封装模型推理能力

说明：

- 当前仓库实际已经是 `React 19`，不再按旧文档中的 `React 18` 约束
- 当前 `react-router.config.ts` 已设置 `ssr: false`，V1 继续走 SPA 路线即可
- `Hash Router` 不是当前发版阻塞项，如 Tauri 环境路由正常，可继续保持现状

### 3.2 推荐库选型

#### 前端基础

- 路由：`react-router`
- 组件：`shadcn/ui`
- 图标：`lucide-react`
- 本地状态：`zustand`
- 异步状态与缓存：`@tanstack/react-query`
- 表单：`shadcn/ui form` + `react-hook-form` + `zod`
- 校验：`zod`
- 图表：`shadcn/ui chart`（底层使用 `Recharts`）
- 拖拽：`@dnd-kit/core`

#### Tauri 插件

- 日志：`tauri-plugin-log`
- 数据库：`tauri-plugin-sql`
- 文件选择：`tauri-plugin-dialog`
- 剪贴板：`tauri-plugin-clipboard-manager`
- 文件系统：`tauri-plugin-fs`
- 应用更新：`tauri-plugin-updater`

#### 数据层

- 运行时数据库访问：`tauri-plugin-sql` 直连 SQLite
- Schema 管理：推荐 `drizzle-kit` + `drizzle-orm/sqlite-core`

说明：

- `Drizzle` 适合做 schema 定义、类型推导和迁移 SQL 生成
- 但在 Tauri WebView 里，不建议把 Drizzle 当成运行时 ORM 主入口
- V1 推荐模式是：`Drizzle 负责建模和迁移`，`tauri-plugin-sql 负责运行时查询`

#### UI 组件原则

- 所有通用 UI 组件优先使用 `shadcn/ui`
- 表单统一使用 `shadcn/ui form` 组件模式，不额外引入新的表单 UI 体系
- 图表统一使用 `shadcn/ui chart`，底层保持 `Recharts`，避免同时维护两套图表方案
- 只有当 `shadcn/ui` 无法覆盖关键交互时，才补充少量专用库

#### AI 与 Agent

- 本地文本模型运行时：`llama.cpp`
- Agent 框架：`LangGraph JS`

说明：

- V1 不做多 Agent，只保留一个 `ContentAgent` 编排入口
- `LangGraph` 作为后续工作流编排基础，比直接堆 if/else 更适合承接 V1.5 的复杂流程
- 不建议 V1 直接引入多 Agent 规划、记忆反思、任务自治，这会显著放大端侧复杂度

### 3.3 不建议在 V1 强绑定的能力

- `SQLite-VSS` 或复杂向量扩展
- 本地视觉模型作为发版硬依赖
- 本地任务调度器 + 精确到每天 8 点的后台执行

原因：

- 这些能力都不是不能做，而是会明显抬高打包、跨平台兼容和故障排查成本
- V1 更适合先跑通核心业务链路，再把重依赖按插件化方式逐步挂上去

## 4. V1 核心架构

### 4.1 分层设计

建议按下面四层拆分：

1. `UI 层`
   负责页面、表单、图表、拖拽、状态展示
2. `应用服务层`
   负责创作、知识库、草稿、截图导入、复盘分析的业务编排
3. `Tauri 命令层`
   负责前端与 Rust 的通信边界
4. `原生能力层`
   负责 SQLite、文件系统、模型调用、日志、更新、备份恢复

### 4.2 目录建议

```text
app/
  components/
  features/
    dashboard/
    creator/
    knowledge-base/
    import/
    analytics/
    settings/
  stores/
  services/
  lib/
  schemas/

src-tauri/
  src/
    commands/
    db/
    models/
    import/
    storage/
    backup/
```

### 4.3 调用原则

- 前端不直接感知模型实现细节
- 所有本地能力通过统一 service 接口访问
- 所有数据库写操作走 service/repository，不在页面里写 SQL
- AI 生成结果必须保留原始输入、版本和时间戳，便于追踪和回滚

## 5. V1 模块设计

### 5.1 工作台首页

V1 首页只保留四块：

- 今日待办
- 最近草稿
- 最近导入
- 数据概览

`每日简报` 在 V1 不做后台定时任务，改为：

- 应用启动时检查今天是否已生成
- 若未生成，则在前台生成一次
- 用户也可手动点击刷新

这样能避免后台常驻、系统权限和跨平台调度问题。

### 5.2 智能创作中心

V1 创作中心保留三栏式结构：

- 左侧：输入需求、人设、风格、参考素材
- 中间：AI 输出区
- 右侧：编辑区、版本记录、插入知识库内容

V1 支持能力：

- 标题生成
- 正文生成
- 正文扩写/缩写
- 标签推荐
- 一键复制
- 自动保存草稿
- 最近 5 个版本回滚

V1 不做：

- 32K 长上下文承诺
- 实时 500ms 风格校准
- 多轮自动自治式内容迭代

### 5.3 截图导入

截图导入是 V1 的 Beta 能力，目标是“可用”，不是“全自动极准”。

V1 支持：

- 拖拽图片
- 从剪贴板粘贴图片
- 选择导入类型：`笔记截图` / `数据截图` / `对标截图`
- 提取后人工确认
- 一键入库到草稿、知识库或数据记录

V1 方案建议：

- 先把截图导入链路做成可插拔 provider
- `provider A`：规则提取 + OCR Beta
- `provider B`：本地视觉模型实验通道

发版策略：

- 截图导入可以发版
- 但视觉模型通道不应成为 V1 发版阻塞项

### 5.4 知识库

V1 知识库优先做强的是“稳定录入 + 快速检索”，不是向量炫技。

V1 推荐：

- 主检索：SQLite `FTS5`
- 辅助排序：`use_count`、`updated_at`、`is_favorite`

V1 支持：

- 分类管理
- 标签
- 搜索
- 插入创作区
- 批量导入 TXT/CSV
- 批量导出

V1.1 再评估：

- `sqlite-vec` 或其他轻量向量方案
- embedding 模型本地化

### 5.5 记忆系统

V1 只做“显式记忆”，不做“自动学习型记忆”。

V1 记忆分三类：

- 账号人设
- 写作偏好
- 禁忌词/禁忌表达

调用方式：

- 生成前拼接进 prompt 上下文
- 用户可在设置页显式编辑和清空

V1 不做：

- 根据用户修改自动学习
- 自动计算记忆权重
- 复杂行为反馈闭环

### 5.6 数据复盘

V1 只做本地记录和轻分析：

- 单篇数据录入
- 周/月聚合图表
- 爆款标记
- 简单 AI 复盘说明

图表实现原则：

- 图表以 `shadcn/ui chart` 为统一外观层
- 只保留对发文提效直接有帮助的图表：发布频率、阅读趋势、爆款占比、标签表现
- 不做大而全的数据看板，避免分析面板喧宾夺主

V1 不做：

- 自动归因模板库
- 排期日历拖拽
- 高级运营洞察

## 7. 模型策略

### 7.1 V1 模型原则

- V1 主打 `单模型文本生成`
- OCR/视觉能力不作为首版稳定性基线
- 先解决“生成质量稳定、调用链可控、资源释放正常”

### 7.2 推荐模型策略

#### 文本生成

- 主模型：`Qwen 系 3B-4B Instruct` 本地量化版
- 运行时：`llama.cpp`

#### 知识检索

- V1 先不用 embedding 做主能力
- 用 `FTS5 + 标签 + 分类 + 最近使用` 解决 80% 需求

#### OCR/视觉

- 作为独立 provider 接口接入
- 先做 Beta，不与主流程强耦合

### 7.3 模型包交付策略

V1 不建议把所有模型直接塞进安装包。

推荐方案：

- 应用安装包保持轻量
- 首次启动时检测本地模型目录
- 若缺失，则引导用户下载模型包
- 模型和应用分离升级

好处：

- 安装包体积可控
- 模型更新不需要整包发版
- 降低代码签名和分发压力

## 8. Agent 方案

### 8.1 选型结论

推荐选 `LangGraph JS` 作为唯一 Agent/Workflow 框架。

原因：

- 生态成熟
- 更适合把生成、检索、校验、回写拆成稳定工作流
- 后续扩展到多步骤编排时，不需要推翻 V1 架构

### 8.2 V1 用法

V1 只保留一个轻量工作流：

1. 收集输入
2. 拼接人设和禁忌词
3. 选择参考知识库
4. 调用本地模型
5. 结构化输出标题、正文、标签
6. 写入草稿和版本记录

### 8.3 V1 不做的 Agent 能力

- 多 Agent 协作
- 自我反思
- 任务自治
- 后台持续运行

## 10. 里程碑拆分

### M1：项目底座

目标：把桌面应用从模板工程变成可持续开发的业务底座。

交付物：

- 页面骨架和导航
- Zustand、React Query、Zod、表单基建
- SQLite 初始化与迁移
- Tauri commands/service/repository 分层
- 设置页和日志能力

验收标准：

- 应用可启动
- 数据库可初始化
- 页面之间可跳转
- 本地配置可读写

### M2：创作闭环

目标：跑通“配置人设 -> 生成文案 -> 保存草稿 -> 回滚版本 -> 复制输出”。

交付物：

- 创作中心
- 草稿列表
- 版本记录
- 本地模型调用链
- LangGraph 单工作流
- 人设和禁忌词配置页

验收标准：

- 用户能完成一次从输入到保存的创作流程
- 失败时有明确错误提示
- 模型调用不会阻塞整个 UI
- 生成结果能稳定复用用户的人设、语气和禁忌词

### M3：知识库闭环

目标：跑通“录入素材 -> 搜索素材 -> 插入创作区 -> 增加使用记录”。

交付物：

- 知识库列表页
- 分类和标签
- FTS5 搜索
- 插入创作区
- CSV/TXT 导入导出

验收标准：

- 1000 条以内素材检索体验流畅
- 搜索结果可直接复用
- 导入失败可定位问题行

### M4：截图与复盘 Beta

目标：让 V1 具备“可演示、可试用”的导入和复盘能力。

交付物：

- 截图拖拽和粘贴
- 导入结果确认页
- 数据录入页
- 周/月图表
- 备份恢复

验收标准：

- 图片能进入导入流程
- 导入结果可人工修改
- 复盘页面能稳定展示本地数据

## 11. 风险与回退策略

### 11.1 最大风险

- 本地模型性能不达预期
- OCR 效果不稳定
- SQLite 向量检索跨平台打包复杂
- 模型包过大导致分发困难

### 11.2 回退原则

- 文本模型优先保住
- OCR 可以 Beta
- 向量检索延后
- 多 Agent 延后
- 模型改为按需下载

### 11.3 发版原则

只要下面四条成立，就可以发 V1：

- 草稿可保存
- 文案可生成
- 知识库可搜索
- 数据可备份

## 13. 最终结论

这个产品最合理的起步方式，不是先证明“端侧一切都能做”，而是先做出一个稳定的本地创作工作台。

V1 成功的标准不是功能最多，而是下面三件事成立：

- 用户愿意长期把草稿和素材放进来
- AI 生成对创作有真实帮助
- 本地端体验稳定，不因为模型和插件链路而频繁出故障

这版方案的核心思想是：

- 用成熟库搭底座
- 用最短链路先闭环
- 把高风险能力做成可插拔，而不是首版刚性依赖
