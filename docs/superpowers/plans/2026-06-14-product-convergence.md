# Neezy Studio 产品收敛与下一阶段实施计划

## 结论

Neezy Studio 的核心方向有差异化，但现在最大问题是产品心智还没收敛：它同时像 ChatGPT、Claude Code、知识库、场景模板工具、人格画像工具、Skill 商店、白板工具。功能都对，但主线还不够锋利。

下一阶段只围绕一条主线收敛：

```text
场景 → 填参数 / 画白板 / 传知识 → 进入专属对话 → 持续迭代 → 沉淀为记忆 / 新场景
```

场景是开始，对话是执行，记忆是沉淀。

---

## 1. 产品定位

Neezy Studio 不应定位为“本地 AI 聊天软件”，而应定位为：

> 面向个人创作者/开发者的本地 Agent 工作室：用场景把复杂任务变成可复用工作流。

当前问题：

- 对话是主入口。
- 场景像模板库。
- 知识、记忆、人格、技能各自独立。
- 用户很难一眼理解“我每天应该怎么用它”。

目标调整：

- 场景成为核心入口。
- 对话成为每个场景的执行空间。
- 记忆/知识成为工作结果的沉淀层。

---

## 2. 信息架构

当前一级导航：

```text
工作台 / 对话 / 场景 / 知识 / 技能 / 画像
```

问题：所有功能权重看起来一样，主线不突出。

建议收敛为：

```text
工作台
场景
对话
知识
设置
```

处理方式：

- 技能放进设置或扩展中心。
- 画像并入知识/记忆中的“关于我”页。
- 画像保留差异化，但不作为日常一级入口。

---

## 3. 场景系统

场景是最值得投入的核心模块。

当前做对了：

- 场景可由 AI 生成。
- 支持动态表单。
- 支持 chip、富文本、思维导图、流程图、白板。
- 场景能进入对话继续迭代。

需要明确边界：

> 场景 = 一组输入组件 + 一段系统任务说明 + 一次专属对话上下文。

这意味着：

- 不再纠结“运行结果页”还是“对话页”。
- 所有场景运行都进入对话。
- 第一条消息由场景生成。
- 后续由用户在专属 session 中持续迭代。

场景卡片应逐步支持：

```text
运行
编辑
复制并改造
查看最近运行
```

核心价值是复用和迭代。

---

## 4. 对话体验

目标对齐 Codex / Claude Code 这类 Agent 工具，而不是普通聊天软件。

当前问题：

- 回复结构仍像普通聊天气泡。
- 工具调用、思考、回复混在一个 timeline 里时容易乱。
- streaming 有时出现上面和下面同时渲染的违和感。
- 场景生成的第一条 prompt 太长，直接展示会污染对话。

建议对话分三层：

```text
用户输入 / 场景上下文（可折叠）
Agent 活动（工具、检索、思考）
最终回复（正文主视觉）
```

场景首条消息技术上完整发送给 agent，但 UI 上显示为：

```text
已启动场景：测试用例生成
[展开查看完整提示词]
```

这样兼顾执行一致性和阅读体验。

---

## 5. 记忆 / 知识 / 人格

当前三套概念：

- 知识：用户显式导入的资料。
- 记忆：对话中抽取的事实/素材。
- 人格画像：从对话归纳的偏好。

用户感知上会混。

建议收敛为两层：

```text
知识库：用户显式导入/保存的资料
记忆：AI 自动沉淀的偏好、事实、历史
```

人格画像不再作为独立数据体系，而作为“记忆的可视化总结”。

### 自研记忆 vs Pi memory package

短期保留自研，暂不直接替换。

原因：

- 当前已经有记忆列表、知识导入、语义搜索、画像 UI。
- 产品需要“可浏览、可搜索、可编辑”的记忆资产。
- Pi packages 里的 memory 包是否暴露稳定的浏览/搜索 API 需要验证。

可评估的 Pi packages：

- `gentle-engram`：跨 session durable memory。
- `pi-hermes-memory`：durable memory、session search、SQLite FTS5、consolidation。
- `@samfp/pi-memory`：保存 corrections/preferences/patterns。
- `@remnic/plugin-pi`：Remnic memory add-on。
- `context-mode`：FTS5 knowledge store + intent lookup。

推荐架构：先抽象 Memory Provider。

```ts
type MemoryProvider = {
  list(): Promise<MemoryItem[]>
  search(query: string): Promise<MemoryItem[]>
  add(input: MemoryInput): Promise<MemoryItem>
  remove(id: string): Promise<void>
}
```

现阶段：

- 自研 SQLite/向量检索作为默认 provider。
- Pi memory package 作为后续可插拔 provider。
- 浏览和搜索 UI 保留在 Neezy 自己这里。

## 6. UI 设计

当前 UI 质感提升了，但个性还没有完全建立。

好的方向：

- 暖白色系比暗色更适合创作工具。
- Sora + Geist 比纯 Geist 有辨识度。
- 毛玻璃可以保留，但要克制。
- 对话页比原来更有层次。

问题：

- 暖色调有一点泛设计工具。
- 卡片、圆角、边框仍有 shadcn 默认味。
- 工作台、场景页、知识页还没统一到“工作室”概念。

建议设计方向：

> Paper Studio / 本地创作桌面：暖白纸张、低饱和琥珀、细线网格、轻量工具感、内容优先。

具体改造：

- 页面背景加轻微纸纹/网格。
- 场景卡片像“工作卡片”，不是普通 card。
- 对话最终回复像文档块，不像聊天气泡。
- 工具调用像终端/任务日志，不像聊天消息。
- 白板场景拥有更强视觉特征。

---

## 7. 功能冗余

高冗余：

- 自研自动记忆 vs Pi memory packages。
- 人格画像单独体系。
- Skill 商店如果只是安装目录，使用频率会低。

中等冗余：

- 场景详情页和场景编辑页重叠。
- 编译预览 / JSON / 表单预览过多，普通用户只需要表单预览 + 高级 JSON。
- 运行场景和普通对话路径曾经混乱，应统一为 sessionStorage 启动链路。

应保留：

- 动态表单。
- 白板 / 思维导图 / 流程图输入。
- 知识库导入。
- 场景生成器。
- 对话 Agent 工具能力。

---

## 8. 下一阶段只做 5 件事

### 1. 打磨场景链路稳定性

目标：

```text
场景表单 → 新 session → 自动发送 → 回复稳定显示
```

验收标准：

- 点击场景运行必定创建新 session。
- URL 只使用 `/chat?session=<id>`。
- 首条场景 prompt 只自动发送一次。
- 页面不闪烁，不需要用户从历史对话重新进入。

### 2. 修好场景编辑体验

目标：

- 编辑时加载旧 JSON。
- 右侧默认真实表单预览。
- JSON 只作为高级入口。
- AI 修改必须基于当前 JSON，不允许换题。

验收标准：

- 打开 `/scenes/designer?edit=<id>` 能看到原场景。
- “选项更细致一点”只改当前场景，不换领域。

### 3. 改造对话为 Agent 工具风

目标：

- 最终回复突出为文档块。
- 工具调用折叠。
- 场景上下文折叠。
- streaming 只呈现一个主输出流。

验收标准：

- 用户能快速区分：上下文、工具活动、最终回复。
- 长 prompt 不污染对话阅读。

### 4. 精简知识 / 记忆 / 画像

目标：

- 画像并入记忆视图。
- 自研记忆保留 UI 与 provider。
- 评估 Pi memory package 作为底层 provider。

验收标准：

- 用户只看到“知识库”和“记忆”。
- 画像成为记忆页里的总结卡片。

### 5. 统一视觉系统

目标：

- 先统一对话页 + 场景页。
- 不全站同时大改。

验收标准：

- 对话页像 Agent 工作台。
- 场景页像可复用工作卡片库。
- 视觉风格统一为 Paper Studio。

---

## 实施顺序

1. 场景启动链路。
2. 场景编辑体验。
3. 对话 Agent 风格。
4. 记忆/画像收敛。
5. 视觉系统统一。

不要并行大改。先把主链路稳定，再改 UI。
