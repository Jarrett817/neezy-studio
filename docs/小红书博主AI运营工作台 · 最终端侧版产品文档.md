# 小红书博主 AI 运营工作台 · 完整端侧产品方案

## 1. 产品目标

Neezy Studio 是一个面向小红书博主的本地优先 AI 运营工作台，帮助用户完成选题、素材沉淀、草稿生成、内容修订、数据录入和复盘迭代。

产品不按 V1/V2 分期描述，直接按完整形态设计。工程实现可以分步骤推进，但任何未接入能力都不能展示假结果。

核心原则：

- 真实能力优先：未接入的模型、采集、OCR、数据库、Agent 不展示假结果。
- 包体精简：应用包不内置大模型，不内置 Hugging Face 下载器，不打包 llama.cpp runner。
- Ollama 管理模型：大模型下载、存储、运行、删除都交给本机 Ollama。
- 本地优先：账号配置、草稿、素材、记忆、复盘数据默认保存在本机。
- Agent 可控：Agent 负责任务编排，关键写入和发布前必须用户确认。
- 低启动成本：界面减少空泛说明、复杂面板和多余分支。

## 2. 用户与交互原则

目标用户：

- 小红书博主
- 内容创作者
- 兼职运营者
- 容易分心、启动困难、对复杂后台工具抗拒的用户

交互原则：

- 每个页面只保留一个最明显的主动作。
- 高级设置、模型参数、Agent 细节默认折叠。
- 空数据、缺模型、下载中、推理中、失败原因都要明确。
- 草稿、导入任务、Agent 运行记录必须能中断后继续。
- 不写心理诊断式文案，不把“专注困难”等标签展示给用户。
- 批量覆盖、删除、入库、发布相关动作必须二次确认或可撤销。

## 3. 核心功能

### 3.1 工作台

- 今日待处理事项
- 最近草稿
- 最近导入任务
- Ollama 与 Agent 状态
- 数据健康状态
- 一键继续上次未完成任务

没有真实数据时只显示空状态和主操作，不展示占位图表、占位待办或占位热门内容。

### 3.2 创作中心

- 选题输入
- 内容目标输入
- 账号人设、语气、禁忌词注入
- 知识库素材引用
- 标题生成
- 正文生成
- 标签生成
- 封面文案建议
- 风险词检查
- 草稿版本保存与回滚
- 一键复制小红书发布格式

默认界面只显示必要输入和主按钮，素材、结构、风格、模型参数放在折叠区。

### 3.3 知识库

- 手动录入素材
- 批量导入 TXT / CSV / Markdown
- 分类与标签
- SQLite FTS5 全文搜索
- 收藏与使用计数
- 插入创作区
- 素材引用溯源

搜索结果必须来自真实数据。

### 3.4 记忆系统

记忆类型：

- 账号定位
- 人设描述
- 语气风格
- 禁忌词
- 高频结构模板
- 已验证有效的标题模式
- 用户手动保存的写作偏好
- Agent 建议写入且用户确认的经验

记忆规则：

- 显式记忆：用户手动创建、编辑、禁用、删除。
- 确认式记忆：Agent 可以建议写入，但必须用户确认。
- 运行时记忆：只在当前任务中使用，不自动写入长期记忆。

### 3.5 Agent 工作流

Agent 是任务编排器，不是聊天装饰。

核心 Agent：

- ContentAgent：选题到草稿生成闭环
- MemoryAgent：检索、整理和建议写入记忆
- ImportAgent：导入截图、链接、文本并结构化
- ReviewAgent：风格、风险词、结构完整性检查
- AnalyticsAgent：基于真实数据生成复盘结论

工作流要求：

- 每一步有明确输入、输出和失败状态。
- 长任务可暂停、重试、恢复。
- Agent 不能静默写入正式数据。
- Agent 输出保存 trace：输入、模型、记忆、工具调用、结果、时间。

Agent 编排使用 LangGraph JS。

### 3.6 数据录入与导入

- 小红书文章链接录入
- 截图拖拽导入
- 剪贴板图片导入
- OCR 提取
- 手动修正
- 结构化字段确认
- 写入知识库、草稿、复盘数据

原则：

- 不托管用户账号密码。
- 需要登录的页面由用户自行登录。
- 如果采集器、OCR、视觉模型未接入，任务必须失败并说明原因。
- 不伪造点赞、收藏、作者、标题、结论。

### 3.7 数据复盘

- 单篇数据录入
- 周/月趋势
- 爆文标记
- 标签表现
- 选题表现
- 发布时间表现
- Agent 生成复盘建议

图表使用 shadcn/ui chart，底层 Recharts。

## 4. 技术架构

### 4.1 技术栈

- 桌面容器：Tauri v2
- 前端：React 19 + React Router 7 + TypeScript
- UI：shadcn/ui + Tailwind CSS + lucide-react
- 状态：@tanstack/react-query + zustand
- 表单：react-hook-form + zod + shadcn/ui form
- 数据：SQLite
- Agent：LangGraph JS
- 本地模型：Ollama

### 4.2 模型管理策略

Ollama 是唯一的大模型管理器。

应用负责：

- 检测 `127.0.0.1:11434` 是否可用。
- 调用 `/api/tags` 获取已安装模型。
- 调用 `/api/pull` 下载模型。
- 调用 `/api/generate` 或 `/api/chat` 执行推理。
- 保存默认模型 ID、运行耗时、错误和输出 trace。

应用不负责：

- 不内置大模型。
- 不内置 Hugging Face 下载器。
- 不内置 llama.cpp runner。
- 不管理模型文件目录。
- 不处理模型删除和磁盘清理，交给 Ollama。

推荐模型：

- 默认轻量：`gemma4:e2b`
- 质量优先：`gemma4:e4b`
- 中文创作备选：`qwen2.5:3b`

### 4.3 性能调度

调度目标：

- 不让模型推理卡住 UI。
- 不在低内存设备上同时跑多个大任务。
- 根据任务类型选择合适模型。

调度策略：

- 小任务：分类、标签、风险词检查，优先使用默认轻量模型。
- 中任务：标题、短正文，使用默认文本模型。
- 大任务：长文重写、复盘总结，使用用户选定的质量模型。
- embedding 与文本生成分开排队。
- OCR / 视觉任务独立队列。

运行状态：

- idle
- downloading
- loading
- running
- throttled
- failed

## 5. 数据设计

核心表：

- app_settings
- accounts
- memory_items
- drafts
- draft_versions
- knowledge_items
- knowledge_item_tags
- imports
- note_metrics
- agent_runs
- model_runs

关键要求：

- 草稿保存版本。
- Agent 每次运行保存 trace。
- 模型每次调用保存模型 ID、输入 hash、输出、耗时、错误。
- 记忆写入必须可撤销。

## 6. UI 设计准则

- 安静、清晰、工作型。
- 不做营销页。
- 不做复杂仪表盘堆叠。
- 不把页面塞满卡片。
- shadcn/ui 能覆盖就用 shadcn/ui。
- 图标用 lucide-react。
- 表单用 shadcn/ui form + react-hook-form。
- 图表用 shadcn/ui chart。
- 不自造通用组件库。
- 一个页面一个主操作。
- 高级选项折叠。
- 空状态不写过度解释。

## 7. 实现顺序

1. SQLite 与 repository 层
2. 账号配置与记忆系统
3. Ollama provider
4. 模型运行记录与调度队列
5. LangGraph Agent 运行记录与 trace
6. 创作中心真实生成闭环
7. 草稿与版本系统
8. 知识库 CRUD + FTS5
9. 导入任务与 OCR provider
10. 数据复盘
11. 备份与恢复

每一步都必须满足：

- 不展示假数据。
- 失败可解释。
- 数据可追溯。
- UI 可继续操作。
