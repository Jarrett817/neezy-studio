# 代码审查约定

在审查或编写匹配文件时，按下列规则检查并给出修改建议；冲突时以仓库既有模式为准。

## 必须遵守的约定

- 禁止抽离只使用了一次的常量。
- 禁止用 `xxx.includes(a)||yyy.includes(a)` 这种写法，用 `includes` 优化。
- 当类型声明明确时，禁止做无意义的类型判断。
- 优先使用环境或者第三方库已经有的 ts 声明，禁止自行生成和猜测。
- 优先使用 lodash 的工具函数优化代码逻辑，使得逻辑精简可读。
- 禁止使用 switch case 去做简单的枚举匹配，用 hash 表的形式优化，使用策略模式优化大量的 if else。
- 禁止在页面上输出类似心理活动、独白类的文案。
- 禁止无意义的函数包装，比如 `function a()`，`function b() { a() }`。
- 适当的拆分组件，同一模块的组件用文件夹分类而不是用 aaa-list、aaa-card。
- 禁止对大量结构相同的重复元素逐个手写多段 JSX；应用数据源配合 `list.map`（或等价遍历）生成，并为列表项提供稳定 `key`。
- 禁止无意义的变量声明，比如 `const a = 0`，`const b = a`。
- 禁止把与入参/上级对象**同构**的字段再逐个抄一遍组装成新对象（例如 `{ videoUrl: input.videoUrl, message: input.message }` 仅为了去掉个别键或做浅清洗）；应使用 `omit` / `pick` / `const { x, ...rest } = input` 等与语义一致的写法，避免无信息增量的中间字面量。
- 使用了 tailwind 的项目必须优先使用 tailwind，除非不得已才使用 css。
- 禁止硬编码的判断，必须使用常量枚举。
- 禁止使用 `==`，必须使用 `===`。

## 审查时的输出方式

- 按严重度标注：必须修改 / 建议优化 / 可选。
- 每条指出文件与大致位置；给出符合上述约定的替代写法要点，避免空泛表述。

## 补充说明

- **`includes` 优化**：多个 `includes` 用 `||` 串起来时，可改为对「候选数组/集合」做一次遍历或 `.some`，避免重复样板。
- **lodash**：优先用已有依赖中的 `lodash` / `lodash-es` 中与场景匹配的 API（如 `groupBy`、`keyBy`、`uniq`、`omit`），避免为单行逻辑强行引入 lodash。
- **类型声明**：优先从依赖包导出类型、`@types/*` 或 DefinitelyTyped；勿手写与官方重复且易错的声明。
- **策略 / 表驱动**：枚举到处理函数用 `Record<Enum, Handler>` 或 `Map`；简单分支仍可读时可保持 `if`，不必为两行逻辑强行抽象。
